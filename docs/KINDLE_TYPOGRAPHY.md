# Kindle Typography â€” Reference for Kanshu

Reference for how Kindle handles typography and what that means for Kanshu's reader. Captures the model we're targeting and the decisions already in place.

## Why this is the right model

Most EPUB CSS is auto-generated by export pipelines â€” InDesign Reflowable Export, Calibre conversions, Pandoc, custom publisher tooling â€” and is bloated, conflicting, and built around print page geometry that doesn't translate to small reflowable screens. A reader that respects every publisher's CSS gives you wildly inconsistent typography book-to-book, broken layouts on small displays, and idiosyncratic colour/spacing choices that distract from reading.

Kindle has been refining its trade-off for 20+ years and lands on a stance that's defensible: **layout is the reader's, typography is mostly the publisher's, until the user says otherwise.** Layout consistency makes the library feel like one place. Letting the publisher's chosen font through gives books their personality.

This doc exists because we already burned ourselves once. Phase 0 shipped with `publisherStyles = false` assuming it would enforce Kindle-style ownership, but it actually stripped the publisher's `font-family` while keeping `@font-face` rules loaded-but-unused. Any character the default Readium font didn't cover rendered as tofu. Fix was flipping to `true`. Knowing the model upfront would have caught that.

## The two-layer model

### Layout â€” always the reader's

Kindle owns and the publisher cannot override:

- **Margins** (Narrow / Standard / Wide, user-selectable). Publisher's `margin` / `padding` on body or html elements gets ignored.
- **Line spacing** (user-selectable). Publisher's `line-height` overridden.
- **Justification** (user toggleable). Publisher's `text-align` may be overridden.
- **Hyphenation** (user toggleable).
- **Column count**. Decided by screen size and orientation.
- **Pagination**. The reader paginates; `@page` rules largely ignored. `page-break-{before,after}` is respected so chapter breaks land where the publisher wanted.
- **Background and text colour**. Themed (white / sepia / dark).

### Fonts â€” publisher default, user override-able

Kindle loads the publication's:

- **`@font-face` declarations** â€” embedded fonts are available.
- **`font-family`** on body / paragraphs â€” applied when "Publisher Font" is selected (the default).
- **`font-style`** (italic) â€” respected; semantic.
- **`font-weight`** (bold) â€” respected; semantic.

When the user picks a Kindle font (Bookerly / Caecilia / Amazon Ember / Palatino / OpenDyslexic / â€¦), that bundled font overrides body text. Embedded fonts targeting specific selectors â€” drop caps via `:first-letter`, fancy chapter titles via class names â€” often still apply because they're scoped where the override doesn't touch.

**`font-size`** is special: Kindle has a user-selectable base size; the publication's absolute sizes are typically ignored, while relative units (em, rem, %) scale against the user's base.

### Less verified, worth testing if they bite us

- Drop caps under font override: Kindle appears to keep publisher's `:first-letter` styling. Readium's `publisherStyles = false` may not preserve it; needs a test book.
- Embedded fonts that don't cover all the publication's characters (e.g. a Latin display font in a Latin/CJK mixed book): Kindle has system fallback fonts; Android WebView has the system font as fallback. Both probably work but worth confirming on the Boox.
- `@media` queries: Kindle evaluates against the reflowable page, not device dimensions. Readium behaves similarly.
- Custom classes / IDs for non-typography properties (chapter title centering, image sizing): respected as-is on both Kindle and Readium.

## Mapping to Readium

Readium 3 exposes the trade-off via `EpubPreferences.publisherStyles`:

| Setting | Effect |
|---|---|
| `publisherStyles = true` | Publisher CSS applies for typography. `@font-face` works. `font-family` and `font-style` stick. Layout settings on `EpubPreferences` (`columnCount`, `pageMargins`, `lineHeight`, â€¦) still wrap around the content; they aren't controlled by this flag. |
| `publisherStyles = false` | ReadiumCSS injects `!important` overrides for typography properties. Whatever we set in `EpubPreferences.fontFamily` / `fontSize` / etc. wins. |

Other relevant `EpubPreferences` fields:

- **Always ours, regardless of `publisherStyles`** (layout-side): `columnCount`, `pageMargins`, `lineHeight`, `letterSpacing`, `wordSpacing`, `textAlign`, `hyphens`.
- **Only set when the user has chosen one** (font-side): `fontFamily`, `fontSize`, `fontWeight`.

## Kanshu's stance

### Phase 0 (today)

- `publisherStyles = true`. Embedded fonts and publisher `font-family` win, so real books render correctly out of the box.
- `columnCount = ColumnCount.ONE`. Layout ownership is partial â€” only column count is enforced right now; the rest of the layout settings come when we build the reader-settings overlay.

### Phase 1+ (when we build reader settings)

Add a settings overlay with font picker, font size, margins, line spacing. Mapping:

| User picks | `publisherStyles` | `fontFamily` | `fontSize` |
|---|---|---|---|
| "Publisher" (default) | `true` | unset | unset |
| Any Kanshu font | `false` | chosen | chosen |

Margins / line height / hyphens always set on `EpubPreferences` regardless.

### What we do NOT want to do

- **Strip publisher CSS entirely.** That kills `@font-face` and produces tofu for non-Latin scripts and fancy typography. We've been there.
- **Let the publisher control layout.** Margins, columns, line height should be Kanshu's no matter what â€” Kindle's biggest win is layout consistency across a library, and we want that.
- **Build per-book overrides.** Kindle doesn't, and the marginal benefit isn't worth the settings complexity.

## Footguns

- `publisherStyles = false` doesn't strip the publisher's stylesheet â€” it injects `!important` overrides for typography properties. Set it without also setting `fontFamily` and ReadiumCSS's default font wins. That font may not cover the publication's script. Tofu.
- `pageMargins` on `EpubPreferences` controls Readium's viewport padding, not the publisher's body margin. With `publisherStyles = true` both apply (Readium's first, then the publisher's inside). Publisher-added margins compound with ours; the reader feels overly padded.
- Non-Latin scripts (CJK, Cyrillic, Hebrew, Arabic) often rely on the publisher's embedded font for proper rendering. Always test with at least one non-Latin book before changing `publisherStyles`.
- Auto-generated CSS from Calibre / InDesign frequently uses `!important` and per-paragraph classes. Even with `publisherStyles = true`, publisher overrides may stick where Readium doesn't expect them. If a book looks visually broken in ways our settings don't explain, suspect publisher CSS first.
